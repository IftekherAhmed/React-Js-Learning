2. Understanding Components

What is a Component?

A component is a reusable, self-contained piece of UI that accepts inputs (props) and returns React elements describing what should appear on the screen.

Analogy: Think of components like LEGO blocks - each block is independent, but you can combine them to build complex structures.

Types of Components in Our Project

1. Layout Components

  • Navbar.jsx - Navigation bar (appears on all pages)
  • Footer.jsx - Footer section (appears on all pages)

2. Page Components

  • Home.jsx - Landing page
  • About.jsx - About page
  • PostListing.jsx - Blog listing with pagination
  • PostDetail.jsx - Single post view
  • CategoryArchive.jsx - Posts filtered by category
  • Search.jsx - Search results
  • Contact.jsx - Contact form

3. UI Components (Reusable)

  • PostCard.jsx - Displays a single post preview (used on 5 pages)
  • HeroSlider.jsx - Hero image carousel
  • SkeletonLoader.jsx - Loading placeholders

What is a Functional Component?

A functional component is a plain JavaScript function that accepts props as an argument and returns a React element (JSX).

Syntax: 
const ComponentName = (props) => {
  // Component logic here
  return <JSX />;  // Must return JSX
};

export default ComponentName;

Example: PostCard Component

From PostCard.jsx 
const PostCard = ({ post }) => {
  return (
    <div className="card">
      <h5>{post.title}</h5>
      <p>{post.body}</p>
    </div>
  );
};

export default PostCard;

What is Destructuring?

Destructuring is a JavaScript feature that allows you to extract values from arrays or objects and assign them to variables in a concise way.

Example: 
// Without destructuring
const PostCard = (props) => {
  const title = props.post.title;
};

// With destructuring (what we use)
const PostCard = ({ post }) => {
  const title = post.title;
};

What is export default?

export default makes a component available for import in other files. Each file can have only one default export.

3. Understanding Context API

What is Context API?

Context provides a way to share data between components without passing props down manually at every level. It's like a global variable for React.

The Problem Context Solves: Prop Drilling

Without Context (Prop Drilling)

<App>
  <Navbar searchQuery={query} onSearch={handleSearch} />
  <MainContent searchQuery={query} onSearch={handleSearch}>
    <Sidebar searchQuery={query} onSearch={handleSearch}>
      <Widget searchQuery={query} onSearch={handleSearch} />
    </Sidebar>
  </MainContent>
</App>

With Context (Direct Access)

<App>
  <SearchProvider>
    <Navbar />     // Access search directly
    <MainContent>
      <Widget />   // Access search directly
    </MainContent>
  </SearchProvider>
</App>

A Provider is a React component that makes context data available to all nested components. Any component inside a Provider can access the context data.

What is Component Reusability?

Component reusability means writing a component once and using it in multiple places with different data.

Benefits:
  • DRY Principle - Don't Repeat Yourself
  • Consistency - Same look across all pages
  • Easy Maintenance - Change once, update everywhere
  • Testable - Test once, works everywhere

6. Routing & Navigation

What is React Router?

React Router enables navigation between "pages" without reloading the browser (Single Page Application). It maps URLs to components.

Routes in Our App

App.jsx 
<Routes>
  <Route path="/" element={<Home />} />
  <Route path="/about" element={<About />} />
  <Route path="/blog" element={<PostListing />} />
  <Route path="/post/:id" element={<PostDetail />} />
  <Route path="/category/:slug" element={<CategoryArchive />} />
  <Route path="/search" element={<Search />} />
  <Route path="/contact" element={<Contact />} />
</Routes>

What is a Static Route?

A static route has an exact path with no variables.

Example: /about always shows the About page

What is a Dynamic Route?

A dynamic route contains variables (using :) that change based on the URL.

Examples:
  • /post/:id - /post/5 shows post with id 5
  • /category/:slug - /category/technology shows technology category

Navigation Methods

What is Link Component?

Link is a React Router component that creates navigation links without page reload (like <a> tag but for SPA).

Usage: <Link to="/about">About Us</Link>

What is useNavigate Hook?

useNavigate is a React Router hook that allows programmatic navigation (navigate from JavaScript code).

Usage: 
const navigate = useNavigate();

// Navigate on button click
const handleClick = () => {
  navigate('/about');
};

What is useParams Hook?

useParams extracts dynamic parameters from the URL (the variables defined with : in routes).

Example: 
// URL: /post/5
const { id } = useParams();  // id = "5"

// URL: /category/technology
const { slug } = useParams();  // slug = "technology"

What is useSearchParams Hook?

useSearchParams reads and modifies the query string in the URL (the part after ?).

Example: 
// URL: /search?q=react
const [searchParams] = useSearchParams();
const query = searchParams.get('q');  // query = "react"

7. Complete Hooks Reference

What is a Hook?

Hooks are functions that let you "hook into" React state and lifecycle features from functional components. They always start with use.

1. useState Hook

useState - Manage Component State

const [state, setState] = useState(initialValue);
What is it? useState is a React Hook that lets you add state variables to your functional components. State is data that changes over time and triggers UI updates when modified.

Why is it necessary?

Without state, your components would be static and couldn't respond to user interactions or data changes. useState makes your app dynamic and interactive.

Key concepts:
  • State is local to the component (only that component can access it)
  • When state changes, React automatically re-renders the component
  • State is preserved between re-renders (doesn't reset)
  • You should never modify state directly - always use the setter function
initialValue
The starting value of the state. Can be any type: string, number, boolean, array, object, or null
state (first value in array)
The current value of the state variable (what you display in your UI)
setState (second value in array)
A function to update the state. Calling this triggers a re-render with the new value
How it works (step by step):
  1. Initial Render: Component renders with the initialValue you provided
  2. User Interaction: Something happens (button click, API response, form input)
  3. State Update: You call setState(newValue) to update the state
  4. Re-render: React automatically re-renders the component with the new state value
  5. UI Updates: The UI reflects the new state without page reload
Example 1: PostListing.jsx - Managing Posts Data
// Three state variables for blog listing
const [posts, setPosts] = useState([]);           // Store all posts (empty array initially)
const [loading, setLoading] = useState(true);     // Show loading spinner (starts as true)
const [currentPage, setCurrentPage] = useState(1); // Pagination (starts at page 1)

// When API returns data:
const data = await response.json();
setPosts(data);        // Update posts state → Component re-renders with posts
setLoading(false);     // Update loading state → Loading spinner disappears

// When user changes page:
setCurrentPage(2);     // Update page state → Component re-renders with page 2 posts

Why needed here? Without useState, we couldn't store the fetched posts, show loading states, or track which page the user is on. The component would have no memory!

Example 2: Contact.jsx - Form State Management
// Form field states
const [name, setName] = useState('');        // User's name input
const [email, setEmail] = useState('');      // User's email input
const [phone, setPhone] = useState('');      // User's phone input
const [message, setMessage] = useState('');  // User's message input

// Feedback states
const [success, setSuccess] = useState(false);   // Show success message
const [error, setError] = useState('');          // Show error message

// When user types in name field:
const handleChange = (e) => {
  setName(e.target.value);  // Updates name state → Input shows new value
};

Why needed here? useState captures user input in real-time. Without it, the form wouldn't know what the user typed, couldn't validate inputs, and couldn't show success/error messages.

Example 3: SearchContext.jsx - Global Search State
// Global search state (shared across components)
const [searchQuery, setSearchQuery] = useState('');       // Current search text
const [searchResults, setSearchResults] = useState([]);   // Search results array

// When user searches:
const updateSearch = (query, results = []) => {
  setSearchQuery(query);          // Update search query
  setSearchResults(results);      // Update search results
};

Why needed here? useState stores the search query and results so both the Navbar (where user types) and Search page (where results show) can access the same data through Context API.

Common Mistake: 
// ❌ WRONG - Direct state mutation
posts.push(newPost);  // This won't trigger re-render!

// ✅ CORRECT - Use setter function
setPosts([...posts, newPost]);  // Creates new array → Triggers re-render

2. useEffect Hook

useEffect - Handle Side Effects

useEffect(() => { // Side effect code (API calls, subscriptions, DOM updates) return () => { /* cleanup code */ }; }, [dependencies]);
What is it? useEffect is a React Hook that lets you run "side effects" - operations that affect something outside the component or need to happen after rendering (like fetching data, setting page titles, or subscribing to events).

Why is it necessary?

React components are meant to be pure - they should just return JSX based on props and state. But real apps need to fetch data, interact with APIs, manipulate the DOM, or set timers. useEffect provides a safe way to do these "impure" operations without breaking React's rendering model.

What are "side effects"?
  • Fetching data from an API (most common in our app)
  • Setting document.title for SEO
  • Setting up subscriptions (like WebSocket connections)
  • Manual DOM manipulation (when React can't handle it)
  • Setting timers (setTimeout, setInterval)
  • Logging/analytics (tracking page views)
Callback Function (first parameter)
The code you want to run after render. This runs after the component renders to the screen.
Dependencies Array (second parameter)
Controls WHEN the effect runs:
  • [] (empty array) - Run once after initial mount only (like componentDidMount)
  • [state] - Run once on mount + every time the specified state changes
  • No array - Run after every single render (rarely needed, can cause infinite loops!)
Cleanup Function (return statement)
Optional: Code that runs before component unmounts or before effect re-runs. Used to clean up subscriptions, timers, or event listeners to prevent memory leaks.
How it works (lifecycle):
  1. Component Renders: JSX is displayed on screen
  2. Effect Runs: After paint, React executes your useEffect callback
  3. Async Operations: Your code runs (API fetch, timer, etc.)
  4. State Updates: When data arrives, you call setState → triggers re-render
  5. Dependency Check: If dependencies changed, effect runs again
  6. Cleanup: If component unmounts, cleanup function runs (if provided)
Example 1: Home.jsx - Fetch Posts on Page Load
useEffect(() => {
  // Define async function inside useEffect
  const fetchPosts = async () => {
    try {
      // Step 1: API request
      const response = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=6');
      
      // Step 2: Convert to JSON
      const data = await response.json();
      
      // Step 3: Update state with fetched data
      setPosts(data);  // Triggers re-render with posts
    } catch (error) {
      // Handle errors gracefully
      console.error('Error fetching posts:', error);
    } finally {
      // Always runs - removes loading spinner
      setLoading(false);
    }
  };
  
  fetchPosts();  // Call the function
}, []);  // Empty array = run ONCE when component mounts

Why needed here? Without useEffect, we'd fetch data during rendering which would cause infinite loops. useEffect ensures data fetching happens after the initial render, and only once (thanks to []).

Example 2: Home.jsx - Set Page Title for SEO
useEffect(() => {
  // Update browser tab title
  document.title = 'BlogHub - Home | Modern Blog Platform';
}, []);  // Run once on mount

Why needed here? This is a side effect because it modifies the DOM outside React (the browser tab). We set it once when the page loads so users see a meaningful title in their browser tab and search engines can index it properly.

Example 3: PostDetail.jsx - Fetch Single Post When ID Changes
const { id } = useParams();  // Get post ID from URL

useEffect(() => {
  const fetchPost = async () => {
    setLoading(true);  // Show loading spinner
    try {
      // Fetch specific post by ID from URL
      const response = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);
      const data = await response.json();
      setPost(data);  // Update post state
    } catch (error) {
      console.error('Error fetching post:', error);
    } finally {
      setLoading(false);
    }
  };
  
  fetchPost();
}, [id]);  // Note: [id] dependency - re-runs when URL changes to different post!

Why needed here? The [id] dependency is crucial! When user clicks from /post/1 to /post/2, the ID changes, useEffect detects this, and fetches the new post. Without the dependency array, it would only fetch once and show the wrong post!

Example 4: Contact.jsx - Multiple useEffects in One Component
// useEffect #1: Set page title
useEffect(() => {
  document.title = 'Contact Us - BlogHub';
}, []);

// useEffect #2: You could add more side effects separately
useEffect(() => {
  // Example: Track page view in analytics
  console.log('Contact page viewed');
}, []);

Why multiple useEffects? It's better to have multiple small useEffects (each doing one thing) than one giant useEffect. This makes code easier to understand, debug, and maintain.

⚠️ Common Mistake - Missing Dependency Array: 
// ❌ WRONG - No dependency array = runs after EVERY render = INFINITE LOOP!
useEffect(() => {
  fetchPosts();
});  // Missing [] causes infinite loop!

// ✅ CORRECT - Empty array = runs ONCE on mount
useEffect(() => {
  fetchPosts();
}, []);  // Runs only once

// ✅ CORRECT - With dependency = runs when dependency changes
useEffect(() => {
  fetchPost(id);
}, [id]);  // Runs when id changes
Dependency Array Cheat Sheet:
Dependencies When It Runs Use Case
[] (empty) Once on mount only Initial data fetch, set page title
[id] On mount + when id changes Fetch data when URL parameter changes
[query, page] On mount + when query OR page changes Search results when filters change
No array After EVERY render Rarely needed (usually a mistake!)

11. API Integration

What is an API?

API (Application Programming Interface) is a way for applications to communicate with each other and exchange data.

JSONPlaceholder API (What We Use)

What is JSONPlaceholder?

JSONPlaceholder is a free fake API for testing and prototyping. It provides mock data that looks like real API responses.

Base URL: https://jsonplaceholder.typicode.com

Endpoint Used In What It Returns

What is fetch()?

fetch() is a built-in browser API that makes HTTP requests to fetch resources (like data from an API).

Returns: A Promise that resolves to the Response object

What is async/await?

async/await is modern JavaScript syntax for working with Promises in a more readable way.

Example: 
// Without async/await (old way - using .then)
fetch('https://api.example.com/posts')
  .then(response => response.json())
  .then(data => setPosts(data))
  .catch(error => console.error(error));

// With async/await (modern way - what we use)
const response = await fetch('https://api.example.com/posts');
const data = await response.json();
setPosts(data);

What is try/catch/finally?

try/catch/finally is error handling syntax that lets you handle errors gracefully without crashing your app.

Structure:
  • try - Code that might throw an error
  • catch - Code that runs if an error occurs
  • finally - Code that runs regardless (success or error)

Complete API Call Pattern

From Home.jsx 
useEffect(() => {
  const fetchPosts = async () => {
    try {
      // Step 1: Make request
      const response = await fetch('https://jsonplaceholder.typicode.com/posts?_limit=6');
      
      // Step 2: Check if response is OK
      if (!response.ok) {
        throw new Error('Network response was not ok');
      }
      
      // Step 3: Convert to JSON
      const data = await response.json();
      
      // Step 4: Update state with data
      setPosts(data);

      // Step 5: Handle errors
      console.error('Error fetching posts:', error);
    } finally {
      // Step 6: Always run (cleanup)
      setLoading(false);
    }
  };
  
  fetchPosts();
